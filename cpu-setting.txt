好问题——这颗 CPU 的具体拓扑（P 核/E 核配置）直接决定并行方案怎么设计。这个型号比较新，我先查一下确切规格。查到了确切规格：270K Plus 是 2026 年 3 月发布的 Arrow Lake Refresh，8 个 P 核 + 16 个 E 核，共 24 核 24 线程（无超线程），与 285K 同构。实测全核负载下 P 核约 5.4 GHz、E 核约 4.7 GHz。这个拓扑对你的方案有两个直接推论：

**推论一：混核"拖后腿"问题比想象中小，不值得复杂绑核。** SubprocVecEnv 是同步栅栏——每个 vec step 等最慢的 worker。但 Arrow Lake 的 Skymont E 核 IPC 很强，与 P 核的实际差距主要是 ~15% 的频率，对 Python 游戏逻辑这类标量负载，E 核 worker 大约只慢 1.2×。栅栏损耗有限，优先级应该放在下面这些更大的收益点上。

**推论二：24 物理核、无 HT，意味着 n_envs 的甜点就是"每核一个 worker"。**

按性价比排序的具体措施：

## 1. 线程钉死的完整版（必做，你机器上这是最大单项）

之前说的 `torch.set_num_threads(1)` 只管 torch；numpy/MKL 的线程池要靠环境变量，而且必须在 spawn 之前设好才能被子进程继承：

```python
# train.py 最顶部，先于一切 import
import os
for v in ("OMP_NUM_THREADS", "MKL_NUM_THREADS", "OPENBLAS_NUM_THREADS"):
    os.environ.setdefault(v, "1")
```

```python
# make_env_fn 的 _init() 内
def _init():
    import torch
    torch.set_num_threads(1)
    return SelfPlayEnv(...)
```

不钉的话是 20 worker × 24 默认线程 = 480 线程在 24 核上互踩，通常拖慢 2 倍以上。

## 2. Worker 常驻 + 对手热替换（Windows 上收益特别大）

Windows 的 multiprocessing 只有 spawn 模式：你现在**每一代**都重建 20 个进程，每个进程重新 import torch/sb3、重新用 openpyxl 解析 xlsx——每代 15~40 秒纯启动开销，50 代累计 15~35 分钟。改成 vec_env 建一次、每代只热替换对手权重：

```python
# SelfPlayEnv 内新增
def set_opponent(self, policy_bytes):
    """热替换对手（经 env_method 调用）。None → 随机对手。"""
    if policy_bytes is None:
        self.opponent = None
        return True
    import torch
    from training.feature_extractor import SplendorFeatureExtractor  # noqa: F401
    self.opponent = torch.load(_io.BytesIO(policy_bytes),
                               map_location="cpu", weights_only=False)
    return True
```

```python
# self_play_loop.py：vec_env 移到 for 循环外只建一次
for generation in ...:
    opponent_policy_bytes = ...          # 序列化逻辑不变
    vec_env.env_method("set_opponent", opponent_policy_bytes)
    ...                                   # 删除每代的 vec_env.close()
```

三个附带说明：换手发生在 episode 中间，每个 worker 每代最多 1 局"上半场旧对手、下半场新对手"，几千局里的 20 局，可忽略；eval 期间 workers 阻塞在管道上零 CPU 占用，只是常驻 ~4-6 GB 内存（你 32 GB 没问题）；`rank` 不再需要乘 generation——env 常驻后 RNG 状态自然演进，种子多样性反而更好。`agent_model.set_env` 也只需在首次调用。

## 3. 一行免费优化：随机对手不需要观测

`_auto_play_opponent` 里 `opp_obs = build_observation(...)` 是无条件执行的，但 `self.opponent is None`（随机）分支根本不用它。把这行移进 `if self.opponent is not None:` 内——整个 gen 0 和之后 5% 的随机对手局白白构建了 203 维观测。

## 4. n_envs 提到 22–24

同步栅栏下吞吐 = n_envs ÷ 最慢 worker 的步时。20 个 worker 摊在 24 核上，栅栏时间已经由 E 核 worker 决定；补满到 24（每核一个）**栅栏时间不变、吞吐 +20%**。但不要超过 24：两个 worker 挤一核会让栅栏时间翻倍，净亏。主进程在 worker 步进阶段本身是空闲等待，与 worker 的 CPU 高峰错开，所以 24 可行，保守取 22。`n_steps=1024 / batch_size=256 = 4`，任意 n_envs 都能整除，无副作用。建议实测：20/22/24 各跑两三分钟，对比 SB3 日志的 `time/fps`。

## 5. 绑核：先测量，别急着做

若要做：Arrow Lake 无 HT，逻辑核=物理核，通常 0–7 是 P 核、8–23 是 E 核（用任务管理器逐核占用确认）。**Windows 陷阱：子进程继承父进程的 affinity mask**——如果先把主进程绑到 P 核再建 vec_env，20 个 worker 会全部挤进 8 个 P 核。顺序必须是先 spawn 后绑主进程，或在各 worker `_init` 里自绑。但如推论一所说，Thread Director 对"24 个同优先级忙进程"的调度已接近每核一个，绑核的预期收益只有个位数百分比，仅在观察到 fps 周期性抖动时值得做。

## 6. 对手推理加速（可选，按迭代次数决定）

各 worker 的单样本 `opponent.predict()` 大概率是单步耗时的主导项——SB3 的 predict 对这个 1.6M 参数小 MLP 而言，obs 转 tensor、分布构造、numpy 回转的固定开销超过矩阵乘本身。轻量版：手写前向（logits → mask 置 -inf → argmax/多项式采样）配 `torch.inference_mode()`，预计对手推理快 2–3×、总吞吐 +20–40%。重量版是集中式批量推理（worker 只跑游戏逻辑，对手观测送回主进程批量过 GPU），~250 行自定义 VecEnv 协议，潜在 1.5–3×——对总时长只有几小时的单次训练不值得，除非你打算围绕奖励函数做很多轮实验。

另提一句备选路线：WSL2/Linux 下 SubprocVecEnv 可用 forkserver，worker 启动近零成本，措施 2 的问题自动消失；但不必为此专门迁移环境。

## 预期与验证

做完 1+2+3+4（合计约 40 行改动），在这颗 CPU 上我预计聚合吞吐 4–8k steps/s：每代 500k 步 ≈ 1–2 分钟训练 + 约 1 分钟评估，50 代全程 **2–4 小时**。验证方法：看 `time/fps`；rollout 阶段任务管理器里 24 核应接近全绿，若整体占用明显低于 90%，说明还有线程配置或栅栏失衡问题。最后一个硬件层面的提醒：250W 睿频上限 + 连续数小时全核负载，散热要能压住，否则 E 核会从 4.7 GHz 往下掉，栅栏时间直接变长——训练负载比游戏更接近烤机。
