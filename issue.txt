**最大的遗漏是"座位不对称"，它同时污染了训练、对手池和评估三个环节，而且藏得很深。**

## 1. 最大遗漏：Agent 永远先手 + `current_player` 特征恒为常数

三个事实叠加在一起：

**事实一：agent 从未以后手身份训练过。** `self_play_loop` 硬编码 `agent_player_idx=0`，而 `SelfPlayEnv.__init__` 里 `starting_player=agent_player_idx`——所以训练中的每一局，agent 都是 P0 且先手。注意即使你以后把 `agent_player_idx` 改成 1，行动顺序也不会变（starting_player 跟着座位走），这两个参数目前是耦合的。

**事实二：观测里的 turn flag 在训练数据中是常数。** `observation.py` 把 `float(state.current_player)` 编进观测。agent 决策点上它恒等于 0——网络对这个输入维度的另一个取值（1）从未见过。而 `evaluate.py` 是换边评估的：一半的局里你的模型坐在 P1 位，喂进去一个训练分布外的特征值，通过 512 宽的 MLP 会产生不可预测的激活偏移。

**事实三（最隐蔽）：对手池里的每个对手，每一局都在分布外运行。** SelfPlayEnv 给对手构建观测用的是 `build_observation(state, opp_idx=1)`，turn flag 恒为 1——但这个对手 checkpoint 当年训练时看到的恒为 0。也就是说你的 self-play 从来不是"打上一代的自己"，而是"打一个被 OOD 特征削弱过的上一代"。典型症状是：训练期对最新对手胜率虚高、ELO 爬升但代际间可能出现非传递性循环、真实棋力提升低于曲线显示。

还有一层策略性后果：在你的 equal-turns 规则下（`check_game_end` 的奇偶逻辑），**所有自然终局都发生在 P1 的行动上**——P0 触发 15 分必然给 P1 留最后一手，P1 触发则立即结束。先手方"冲分要预留对手一次反击"、后手方"可以直接绞杀收局"是两套不同的终盘战术，你的 agent 只学过前者，评估和实战却各有一半要求后者。

**修法**（改动很小，`SelfPlayEnv.reset` 里"对手先行"的分支已经写好了，只是从未被触发）：

```python
# self_play_env.py: 座位与行动顺序解耦，P0 恒先手
self.inner = SplendorEnv(self.cards, render_mode=render_mode,
                         starting_player=0, max_turns=max_turns)

# self_play_loop.py: 一半环境让 agent 坐 P1
make_env_fn(..., agent_player_idx=i % 2, ...)
```

这样 turn flag 的两个取值都进入训练分布（它反而变成有用的先后手信号），对手也各有一半局面回到自己的训练分布内。注意：修完之后池子里的旧 checkpoint 是在旧分布下训出来的，最干净的做法是重开训练；至少要意识到过渡期的 ELO 不可比。

## 2. 其他你可能没意识到的点（按优先级）

**终局奖励的实际路径与你想的不同。** 由于上面"终局总在 P1 行动上"这一事实，当前配置下 agent（P0）的终局奖励**从不**走 `rewardcc.py` 的 terminal 分支，而是全部经由 `SelfPlayEnv.step` 里 `reward = -opp_reward` 的替换路径。这条路径能与你的 PBRS 收口恰好等价，完全依赖 Φ 的严格反对称性（Φ_opp = −Φ_agent）——一旦你给 Φ 加任何非相对项，负号替换就悄悄失效。另外那里的注释写着 "+1/-1/0"，与实际的 ±10 已经不同步，说明这两个模块出现过漂移；还有一个细节：替换是覆盖而非叠加，agent 制胜一手的 dense 塑形被丢弃了（影响不大，但要知道）。建议加一行 assert 或注释把这个耦合钉死。座位随机化之后，两条终局路径都会被真实走到，量纲一致就更重要了。

**截断局赢家已判但奖励未发。** `env.py` 的顺序是 `compute_reward` 在前、"truncated 强制 game_over 并判 winner"在后——达到 200 回合上限的局，双方都只拿到 dense 奖励，±10 从未发放；若截断发生在对手回合，agent 还会拿到"负的对手 dense 奖励"这种噪声替换值。把强制终局的判定挪到 `compute_reward` 之前即可。200 回合上限下这很罕见，但"摆烂拖满"的策略目前不会被结局惩罚。

**对手池的 ELO-softmax 温度退化。** `np.exp(elos_centered)` 直接对 ELO 原始分差做 softmax：10 分差距就是 e^10 ≈ 22000 倍权重，这 40% 的分支实际上永远选 ELO 最高那一个。你以为的采样分布是"50% 最新 + 40% 按强度加权 + 10% 均匀"，实际是"50% 最新 + 40% 榜一 + 10% 均匀"。除以一个温度（ELO 尺度上 T≈50~100）才是你想要的效果。

**gen 0 之后再也没有随机对手。** `random_opponent_prob` 这个名字有误导性——它是"从池中均匀抽一个 checkpoint"，不是"随机动作对手"。如果你想保留一点对随机/弱策略的鲁棒性保险，需要显式加一个真正的随机对手分支。

**评估的统计功效偏低，且换边未配对牌堆。** `cheap_eval_games=15` 对 = 30 局，胜率的标准差约 ±9%，代际 win_rate 曲线会很毛糙（K=16 缓冲了 ELO，但别过度解读单代数字）。另外换边的两局用了不同 seed（`match_idx` vs `match_idx+100000`），牌堆不同，换边并没有真正对消运气——同一对局用同一 seed 镜像开局，方差会小很多。

**一个一致性小事**：`PPO_CONFIG["gamma"]` 与 `SHAPING_GAMMA` 目前数值上都是 0.99 但没有代码级绑定，且 `env.py` 调 `compute_reward` 时没传 gamma（用的默认值）。呼应我们上次讨论的：让 config 成为唯一事实来源，把它传下去。

如果只做一件事，就先修座位不对称——它是唯一一个会系统性歪曲你整个 self-play 学习信号的问题，其余都是二阶修正。修完后建议用 `--test` 跑一次，并在日志里按座位分别统计胜率（agent-as-P0 vs agent-as-P1），确认两个分布都在被学习。
