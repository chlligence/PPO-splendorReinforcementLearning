

## 二、剩余问题（按重要性）

**1. 评估里的 `starting_player = match_idx % 2` 现在反而制造了分布外对局，建议删除。** 修完座位解耦后，训练中 P0 永远先手：agent 坐 P1 时永远是后手。但 evaluate.py 保留了旧的先手轮换——奇数场次让 P1 先动。于是一半的评估局是"坐在 P1 位却先手"，这个组合在任何一方的训练分布里都不存在（turn flag 与行动顺序的相关性被打破）。双方同等受损所以相对胜率大致公平，但它给评估注入了无谓的 OOD 噪声。换座+同 seed 本身已经完全抵消了先手优势，直接固定：

```python
starting_player = 0   # 两处 reset 后的覆盖逻辑随之可整段删除
```

**2. Worker 的 CPU 线程超订，会实际拖慢吞吐。** 20 个子进程各自的 torch 默认抢全部 24 核做单样本前向，典型的 SubprocVecEnv 性能陷阱。在 `make_env_fn` 的 `_init()` 里加两行：

```python
import torch
torch.set_num_threads(1)
```

**3. 评估 seed 集每代复用（0..39 固定牌堆）。** 好处是代际胜率严格可比、方差小；代价是所有 ELO 都建立在同 40 副牌上。这是可辩护的取舍，但要自知：它衡量的是"这组牌上的相对强弱"。若想要独立采样，改 `match_seed = generation*100_000 + match_idx`，接受代际间多一点噪声。

**4. 规则保真度的两个天花板（可能是有意设计，点出来供确认）：** 动作空间没有"拿 2 个不同色"——真实规则里场上只剩 2 色时允许，且 8 币时"只拿 2 个避免弃币"是真实技巧，你的 agent 只能拿 3 后被启发式代弃；弃币本身也由固定启发式代劳（真规则可自选、包括弃金）。这两点限制的是对真人对局的迁移上限，不影响自洽训练。

**5. 杂项：** `OpponentPool.elo_prob` 参数已死（实际概率是剩余的 0.35，与存储的 0.40 不符）；`estimate_elo` 文档说 top 3 实取 top 2；`evaluate_generation` 里 `MaskablePPO.load(latest_entry.path)` 未指定 device（会上 GPU，无害但与别处不一致）；rewardcc 内部的 `SHAPING_GAMMA` 常量成了被绕过的残留默认值。最重要的一条纪律：**旧 checkpoint 和旧 pool_index 全部作废，清空重训**——它们是在"恒 P0+flag恒0"分布下训出来的，混入新池会污染前几代。
